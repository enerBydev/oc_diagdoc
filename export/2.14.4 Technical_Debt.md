# 2.14.4 Technical Debt

> La deuda técnica es como la deuda financiera: útil si se usa estratégicamente, mortal si se ignora.

---

## Clasificación de Deuda

| Tipo | Origen | Acción |
|------|--------|--------|
| **Prudente/Deliberada** | "Salgamos rápido ahora, refactorizamos luego" | Documentar + Ticket Jira |
| **Imprudente/Deliberada** | "No tenemos tiempo para tests" | ❌ Inaceptable |
| **Prudente/Inadvertida** | "Aprendimos una mejor forma de hacerlo" | Refactor progresivo |
| **Imprudente/Inadvertida** | Incompetencia o falta de conocimiento | Capacitación urgente |

---

## Estrategia de Pago

No dedicamos "Sprints de Refactorización" (casi nunca funcionan). Usamos la **Regla del Boy Scout**:

> "Deja el código mejor de lo que lo encontraste".

Al tocar un archivo para una feature:
1. Arregla 1 deuda pequeña (ej. tipos any, función larga).
2. Si la deuda es grande, crea un ticket `TECH-DEBT`.

---

## Tracking

Usamos comentarios especiales en el código:

```typescript
// TODO(debt): Esta función es O(n^2), refactorizar a Map O(n) cuando usuarios > 1000
// FIXME: Parche temporal para bug de Safari iOS 15
```

Estos se escanean en CI y generan un reporte.

---

## Cuadrante de Priorización

| Impacto | Esfuerzo | Prioridad |
|---------|----------|-----------|
| Alto | Bajo | **Inmediata (Quick Win)** |
| Alto | Alto | **Planificar (Epic)** |
| Bajo | Bajo | **Boy Scout Rule** |
| Bajo | Alto | **Ignorar (Por ahora)** |
