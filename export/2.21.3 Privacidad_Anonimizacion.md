# 2.21.3 Privacidad y Anonimización

> **[FUTURO]** Análisis de datos ético y legal, protegiendo la identidad individual.

---

## Principio: Data Minimizada

El equipo de Data/Analytics NO necesita saber *quién* es el usuario, solo *qué hizo*.

### Tratamiento de PII (Personally Identifiable Information)

Antes de que los datos toquen el Data Warehouse, campos sensibles deben ser ofuscados.

| Campo | Tratamiento | Resultado |
|-------|-------------|-----------|
| `email` | SHA-256 Hash + Salt | `a3f5...9z1` |
| `telefono` | Masking | `******1234` |
| `nombre` | Drop / Null | `NULL` |
| `direccion_exacta` | Geohash (Precision 5) | `9q9hv` (Área general) |
| `tarjeta_credito` | Tokenized (Vault) | `tok_visa_123` |

---

## Control de Acceso (RBAC)

Aunque los datos estén anonimizados, el acceso al Warehouse es estricto.
1.  **Data Engineers:** Acceso total a Raw y Staging.
2.  **Data Analysts:** Acceso solo a tablas transformadas (Mart).
3.  **Business Users:** Acceso solo a Dashboards visuales (Metabase).

---

## Cumplimiento (Right to be Forgotten)

Si un usuario solicita borrar sus datos:
1.  Se borra de Supabase (Operativo).
2.  El pipeline de Airbyte debe propagar el `DELETE` o marcar el registro como `soft_deleted` en el Warehouse para excluirlo de reportes futuros.
