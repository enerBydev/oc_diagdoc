# 2.17.1 Presupuesto Core Web Vitals

> Límites numéricos estrictos para la salud técnica del proyecto.

---

## Tabla de Presupuestos (Budgets)

Estos límites se configuran en `lighthouserc.json` y rompen el build si se exceden.

### Métricas de Usuario (RUM)

| Métrica | Límite Aceptable | Límite Crítico | Acción si excede |
|---------|------------------|----------------|------------------|
| **LCP** (Largest Contentful Paint) | < 2.5 s | > 4.0 s | Optimizar imágenes/Fuentes |
| **FID** (First Input Delay) | < 100 ms | > 300 ms | Reducir Main Thread JS |
| **CLS** (Cumulative Layout Shift) | < 0.1 | > 0.25 | Reservar espacio (Skeleton) |
| **INP** (Interaction to Next Paint) | < 200 ms | > 500 ms | Optimizar Event Handlers |

### Métricas de Recurso (Build)

| Recurso | Presupuesto (Gzipped) | Estrategia de Reducción |
|---------|-----------------------|-------------------------|
| **JS Initial Bundle** | < 150 KB | Code Splitting, Tree Shaking |
| **CSS Initial** | < 30 KB | PurgeCSS, Tailwind Minify |
| **Imagen Hero** | < 50 KB | WebP/AVIF, Responsive sizes |
| **Fuentes (Google Sans Flex)** | < 40 KB | CDN Google, Subsetting WOFF2 |

---

## Estrategias de Corrección

1.  **Lazy Loading:** Componentes fuera del viewport (ej: Footer, Modales) deben cargarse dinámicamente (`defineAsyncComponent`).
2.  **Optimización de Medios:** Ver [[Proyecto OnlyCarNLD/Datos/2.15. Arquitectura_Media]] para políticas de compresión automática.
3.  **Terceros:** Scripts de analítica o chat deben cargarse con `delay` o `interaction` strategy en Nuxt.
