## Archivo Maestro: `seed.sql`

Supabase ejecuta automáticamente `supabase/seed.sql` después de un `db reset`.
Sin embargo, separamos la lógica por ambiente.

### 1. Catálogos Base (Prod y Dev)
Datos que SIEMPRE deben existir (Roles, Tipos de Servicio, Configuración Inicial).

```sql
-- roles
INSERT INTO auth.roles (role) VALUES ('admin'), ('operador'), ('cliente');

-- servicios base
INSERT INTO public.servicios (nombre, precio_base) VALUES ('Lavado Premium', 250);
```

### 2. Datos Dummy (Solo Dev)
Para desarrollo, necesitamos usuarios, transacciones y mensajes falsos para probar la UI.
Usamos un script de Python `generate_faker_seed.py` para generar miles de registros SQL o usamos Snaplet.

---

## Sanitización de Datos (Staging)

Para el entorno de **Staging**, a veces copiamos datos de Producción, pero debemos **anonimizarlos**.
- Emails -> `user_123@example.com`
- Teléfonos -> `555-000-0000`
- Nombres -> `Test User`

*Nunca copiamos datos personales reales (PII) a entornos de desarrollo inseguros.*
