# 2.7.2 Git Workflow

> Flujo de trabajo para colaboración y versionado.

---

## Estrategia de Ramas

Utilizamos un modelo simplificado enfocado en **Continuous Delivery**.

- **main:** Rama de producción. Siempre desplegable. Protegida.
- **feat/**: Nuevas funcionalidades (ej. `feat/login-google`).
- **fix/**: Corrección de bugs (ej. `fix/calendar-timezone`).
- **chore/**: Mantenimiento, dependencias (ej. `chore/update-nuxt`).

### Reglas
1. Nunca hacer commit directo a `main`.
2. Todo cambio entra vía Pull Request (PR).
3. PR requiere build exitoso (CI) para mergear.

---

## Conventional Commits

Formato obligatorio para mensajes de commit:
`<tipo>(<alcance>): <descripción>`

| Tipo | Uso | Ejemplo |
|------|-----|---------|
| **feat** | Nueva funcionalidad | `feat(auth): add google login` |
| **fix** | Corrección de bug | `fix(calendar): resolve timezone error` |
| **docs** | Documentación | `docs(readme): update install implementation` |
| **style**| Formato (espacios, etc) | `style(ui): run prettier` |
| **refactor** | Código sin cambios lógicos | `refactor(store): simplify auth logic` |
| **test** | Añadir pruebas | `test(unit): add user store tests` |
| **chore** | Tareas de build/deps | `chore(deps): upgrade nuxt to v4` |

---

## Pull Requests (PR)

Plantilla básica para descripción de PR:

```markdown
## Tipo de Cambio
- [ ] Feature
- [ ] Bugfix

## Descripción
Explicar qué hace este cambio y por qué.

## Testing
Pasos para verificar que funciona.
```
